WebAutomation-Impel - Approach Document
=======================================


1. Overview
-----------

This document outlines the architecture, design decisions, and implementation details of the WebAutomation-Impel framework -- a Maven-based Java web automation solution built with Microsoft Playwright and TestNG.

The framework automates end-to-end testing for a web application that includes user login, home page navigation, and complaint creation workflows. It follows industry-standard design patterns to ensure maintainability, reusability, and scalability.

4. Design Patterns Used
-----------------------

4.1 Page Object Model (POM)

Each web page is represented by a dedicated Java class that encapsulates:

- Locators -- stored as private final String fields at the top of the class.
- Actions -- public methods that interact with the page elements.
- Page reference -- a Playwright Page object passed via the constructor.

Benefits:
- Single point of maintenance when UI changes.
- Clean separation between test logic and page interaction.
- Reusable methods across multiple test classes.

Example (LoginPage.java):

    public class LoginPage {
        private final Page page;
        private final String usernameField = "input[name='username']";

        public LoginPage(Page page) {
            this.page = page;
        }

        public void login(String username, String password) {
            page.fill(usernameField, username);
            // ...
        }
    }

4.2 Base Test Pattern

A shared BaseTest class annotated with @BeforeMethod and @AfterMethod handles:

- Browser lifecycle management (create/destroy per test method).
- Common pre-conditions (navigation, login).
- Shared protected fields accessible by all test subclasses.

Benefits:
- Eliminates duplicate setup/teardown code across tests.
- Guarantees every test starts with a clean browser session.
- Centralizes configuration reads.

6. Execution Flow
-----------------

Step 1      Maven / TestNG triggers test execution
                |
Step 2      testng.xml selects test classes to run
                |
Step 3      @BeforeMethod (BaseTest.setUp)
                |
                +-- 3a. Playwright.create()
                +-- 3b. Launch Chromium (Chrome channel, headless from config)
                +-- 3c. Open new page, set 60s timeout
                +-- 3d. Navigate to base.url
                +-- 3e. Auto-login via LoginPage
                |
Step 4      @Test method executes
                |
                +-- Instantiate required Page Objects
                +-- Perform page interactions
                +-- Assert expected outcomes
                |
Step 5      @AfterMethod (BaseTest.tearDown)
                |
                +-- 5a. Thread.sleep(10000) -- observation window
                +-- 5b. Browser.close()
                +-- 5c. Playwright.close()
                |
Step 6      TestNG generates report

8. How to Run Tests
-------------------

8.1 Maven CLI

    Run all tests:
        mvn clean test

    Run a specific test class:
        mvn clean test -Dtest=LoginTest

    Run using testng.xml suite:
        mvn clean test -DsuiteXmlFile=testng.xml

8.2 IDE (IntelliJ IDEA / Eclipse)

    1. Right-click on a test class or method.
    2. Select Run (TestNG will be auto-detected).
    3. Or right-click testng.xml and select Run.

8.3 Via testng.xml

Run the testng.xml file directly to execute the configured suite. To include additional tests, add class entries:

    <classes>
        <class name="tests.LoginTest"/>
        <class name="tests.HomeTest"/>
        <class name="tests.ComplaintTest"/>
    </classes>

10. Best Practices Implemented
------------------------------

- Page Object Model -- UI locators and actions separated from test logic.
- Single Responsibility -- Each page class handles one page; each test class handles one feature.
- Centralized Configuration -- All environment values in config.properties, read via ConfigReader.
- Inherited Setup/Teardown -- BaseTest eliminates duplicate lifecycle code across all tests.
- Auto-Login Before Tests -- Login is handled in @BeforeMethod so tests start from an authenticated state.
- Clean Browser Per Test -- Each @Test gets a fresh Playwright + Browser instance, preventing state leakage.
- Explicit Waits -- waitFor() used before visibility checks instead of arbitrary sleeps.
- Null-Safe Cleanup -- tearDown() checks for null before closing browser and Playwright.
- Thread Interrupt Handling -- InterruptedException properly restores the interrupt flag.
- Locators as Constants -- Stored as private final String fields for easy maintenance.
